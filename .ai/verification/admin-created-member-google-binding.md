# Admin-Created Member Google Binding Verification

- decision: approve
- release_target_supported: local_dev

## Behavior And BDD Alignment

The implemented behavior matches the approved Behavior Spec for local_dev:

- Admin member management remains on `/settings/members`.
- Non-admin members are redirected away from member management and cannot access create/link/edit actions.
- Admins can create unbound members with app-owned display name and role selection.
- Blank display-name validation returns field-level `FieldError` copy.
- Member rows expose the expected statuses: `未綁定`, `待綁定`, `已失效`, `已綁定`, and `已停用`.
- Bound and disabled members hide binding actions.
- One binding modal covers unbound link generation, active-link re-copy, and expired-link regeneration.
- `/members/bind` validates missing, invalid, expired, used, and valid token states without exposing household data.
- Google sign-in start preserves the binding token through `/members/bind/callback`.
- Binding acceptance updates the pre-created member instead of creating an unplanned member.
- Already-linked Google accounts, stale bound-member tokens, disabled targets, and invalid targets are rejected by domain/adapter tests.

## Technical Design Alignment

- `Member.status = invited` is reused for unbound members and rendered as product copy `未綁定`.
- New generated links use `/members/bind?token=...`; legacy `/invite/accept` remains only as compatibility code and is not generated by this slice.
- `MemberInvitation` is reused as the binding-token model, with new behavior requiring `memberId` for generated binding links.
- Pending links are reused until expiry; expired pending links are revoked before replacement.
- Binding acceptance runs member activation and invitation acceptance inside a Prisma transaction.
- Token validation uses `tokenHash`; admin re-copy uses encrypted token fields and `MEMBER_BINDING_TOKEN_ENCRYPTION_KEY`.
- Server actions own access checks and return typed action states; UI components keep rendering/dialog/clipboard responsibilities.
- The implementation stays within the existing Next.js, Prisma, Better Auth, Vitest, Playwright, Tailwind, and shadcn-style component foundation.

## Prototype Gap Closure

- Prototype-only local state in `/settings/members` was replaced with server-loaded member-management data.
- Prototype-only preview binding links were replaced with persisted encrypted binding tokens.
- Simulated binding completion was replaced with `/members/bind/callback` and identity-access adapter behavior.
- Query-state-only `/members/bind` errors were replaced with database-backed token validation.
- Fixture-only member states are now covered by E2E seed data and browser assertions.
- Production email delivery remains out of scope and is still accepted for local_dev; links are copied manually.

## Domain Verification

The domain delta is reflected in implementation:

- Membership is admin-created before Google identity binding.
- Binding links are member-specific, not account-agnostic invitations.
- App-owned member display name and role remain authoritative after binding.
- Google identity is linked to exactly one existing active member.
- Invalid, expired, used, disabled-target, already-bound-target, and already-linked-account paths do not grant household access.
- The generic invitation-created-member path is no longer exposed in active member management UI.

## Residual Risks

- Production readiness is not implied. Production still needs target-aware assessment for secrets, OAuth callback URLs, migration rollout, rollback, monitoring, and smoke checks.
- `MemberInvitation.memberId` remains nullable for legacy compatibility; future hardening can make it non-nullable after old invite rows are migrated or retired.
- Real Google OAuth was not exercised in automated tests; this is acceptable for local_dev because the spec requires automated tests to stay independent from real OAuth credentials.
