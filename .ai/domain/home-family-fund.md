# Home Family Fund Event Storming

## Delivery Profile
This artifact inherits `mvp` delivery and `local_dev` release target from `idea-home-family-fund`. Discovery is constrained to a single household web app where all functional pages require login, core workflows work on desktop and mobile, and payment execution, bank sync, and production hosting concerns remain outside the first release.

## Policies
| When Event Happens | Policy / Rule | Command Issued | Notes |
|---|---|---|---|
| Member attempts to access functionality | All functional pages require Google sign-in and app-owned member authorization. | Authenticate member | Google proves identity; the app decides household membership, roles, and capabilities. |
| Admin needs to add a household participant | MVP membership starts with an admin-created member record. A Google account cannot create an unplanned household member by signing in or accepting a generic invite. | Create member | Admin owns display name, initial role, and access status before the real user binds Google. |
| Admin needs the real user to claim an admin-created member | Binding is done through a member-specific binding link tied to one unbound member record. | Generate member binding link | Link expiry, regeneration, revocation, and reveal/copy behavior are product policy decisions for the active slice. |
| Google account signs in through a binding link | The Google identity can bind only to the link's unbound member if the link is valid and the Google account is not already bound to another member. | Bind Google account to member | Invalid, expired, reused, disabled-member, wrong-state, and already-bound-account cases must not expose household data. |
| Google account signs in after binding | Google profile name and image can initialize display name and avatar, but app-owned membership decides access. | Authenticate member | Display name becomes app-owned after initialization; avatar remains Google-owned for this slice. |
| Admin updates member profile | Admin can update display name only; admins cannot edit avatars. | Update member display name | The updated display name is visible to all users wherever member names appear. |
| Member wants to edit own profile | Self-service display-name/avatar editing is deferred from this slice. | None in MVP slice | Profile management remains admin-only for display name; avatar remains Google-sourced. |
| Member logs out | Logout ends the Better Auth/app session and returns to an unauthenticated sign-in surface. | Log out | Enables switching Google accounts and verifies invitation/wrong-account states. |
| Member permissions changed | New permissions determine allowed commands immediately. | Re-evaluate authorization | Admin is the only role that changes permissions in MVP. |

| Ledger record is deleted | Deletion means voiding the record for MVP local_dev: active reports, record lists, category summaries, and reimbursement tables exclude it, but its identity is retained for audit trace and existing relational references. | Delete ledger record | Hard deletion is deferred because reimbursed expenses can be referenced by reimbursement batches and because financial corrections benefit from traceability. |

| Reimbursement payment rows are uploaded | Direct CSV import of reimbursement payment evidence is not supported in the active slice because external rows cannot safely identify the exact expenses they settle. | None in this slice | A future reconciliation workflow may upload payment rows, present candidate expenses, require manual linking, and then record payment evidence. |

| Record detail reimbursement submitted | The actor must provide payment details and confirm before the selected expense is marked reimbursed. | Confirm record detail reimbursement | Confirmation protects against accidental one-record settlement from the detail dialog, and the payment evidence explains the real-world money movement. |
| Reimbursement payment is recorded | Payment evidence must match the reimbursed expense total for the paid-to member and household, and it must be linked to the reimbursement batch. | Record reimbursement payment | The payment record is audit evidence, not a new income or expense ledger record, so monthly income/expense totals are not double-counted. |

## Risks and Open Questions
- Primary UI locale is decided as Traditional Chinese (`zh-TW`); currency remains unresolved, with TWD likely for MVP.
- Role composition is unresolved: admin and finance manager may need to be independent roles that one member can both hold.
- Category management permission is decided for MVP: only admins can see the category sidebar entry, browse category management, create categories, rename categories, or archive categories. `manage_categories` capability is dormant for this workflow unless future delegation is approved.
- Category visual identity and ordering are decided for MVP: admins can configure active categories with approved palette colors, approved Lucide-backed icons, and manual order within each category type; new categories append to their type by default; archived categories retain visual identity for history but do not participate in new-record ordering.
- Record query rules are decided for the active search/filter slice: keyword search covers approved visible record fields; sorting supports newest first, oldest first, highest amount first, and lowest amount first; category filtering is limited to active categories; archived categories are not searchable/filterable; date filters use occurrence dates and support start-only, end-only, or bounded ranges; record queries do not change mutation permissions.
- Record query language is still being refined for UI: the member/fund participation filter must let users find records involving a selected member or the household fund across income and expense records without forcing separate income-source and expense-payer controls.
- Reimbursement-status query handling is partly unresolved: users need `已退款` and `未退款` filters, but the behavior spec must decide how income and fund-paid expenses appear when such a filter is active.
- Recurring-rule management permissions remain unresolved; current artifact marks them as admin or authorized manager.
- Finance manager delete permission is decided for MVP: finance managers cannot delete other members' records. Admin-managed permission expansion may allow this later if explicitly enabled.
- Member onboarding is narrowed for the active slice: admins create member records first, then generate member-specific binding links for real users to bind Google accounts. Generic invitation links that create unplanned members are no longer the intended product direction.
- Binding-link policy remains open for the active slice: status naming, expiry, one-time reveal versus re-copy, regeneration, revocation, and replacement route naming must be decided before implementation.
- Display-name ownership is decided for the active slice: Google profile can provide the default, but app-owned display name is what everyone sees, and admins can edit display names.
- Avatar ownership is decided for the active slice: Google profile provides the avatar, and admins cannot edit avatars. Whether avatar syncs every login or is copied once remains open.
- Self-service profile editing is deferred from the active slice; non-admin members cannot edit their own display name in this MVP path.
- Logout is required for the active slice so users can end sessions and switch Google accounts; exact UI placement and return route remain UX decisions.
- Reminder delivery is unresolved; MVP can use in-app pending reminders unless external notification is selected later.
- Expense split rules are unresolved; MVP assumes one category and one upfront payer unless changed.
- Reimbursement payment-flow capture is decided for the active slice: reimbursement requires payment evidence, but the app does not execute an external transfer and does not create a second ordinary expense in monthly income/expense totals. MVP assumes one reimbursement batch records one payment from the household fund to one paid-to member; paid-to member, amount, and payment source are derived facts rather than editable form fields. Split payments, partial reimbursement, and reversal/correction remain future policy questions.
- Reimbursement payment search is introduced as read-only discovery through Reporting: `退款金流紀錄` may identify reimbursement payment evidence, but these results remain separate from ordinary ledger records, are not selectable for batch delete/refund in this slice, and do not affect income/expense totals. Prototype and Behavior Spec must decide mixed result versus grouped result versus explicit result-type filter.
- CSV import is introduced as a controlled bulk command path for ledger records, not a separate source of financial truth. Open decisions remain for CSV column contracts, member/category matching, duplicate detection, all-or-nothing versus partial-success commit semantics, and import history. Direct reimbursement payment import is deferred; a future reconciliation workflow would need manual matching from payment rows to exact member-paid expenses.
- Record detail reimbursement wording is decided for the active slice: the button label is `退款`, and it marks an eligible member-paid expense reimbursed after confirmation; it does not execute an external payment transfer.
- Deletion semantics are unresolved; hard delete is simpler, but void/archive may better protect financial history.
- RWD acceptance needs concrete workflow priority: browse, create, monthly report, reimbursement, and member management may need different mobile density.
